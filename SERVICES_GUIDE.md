# 📖 Руководство по созданию сервисов в rProxy

Это подробное руководство описывает каждый шаг процесса публикации ваших локальных сервисов в интернет через rProxy.

---

## 🚀 Общий путь создания любого сервиса

Процесс создания сервиса в rProxy состоит из следующих этапов:

1.  **Определение цели**: Вы указываете локальный IP-адрес и порт устройства в вашей сети (например, `192.168.1.1:80` для роутера или `192.168.1.10:8123` для Home Assistant).
2.  **Выбор метода доступа**:
    *   **Через домен (порт 443)**: Красивый адрес вида `my.domain.com` с автоматическим SSL (HTTPS).
    *   **Через порт VPS**: Доступ по IP-адресу сервера и выбранному порту (например, `1.2.3.4:26001`).
3.  **Генерация конфигурации Nginx**: Скрипт создает файл настроек, специфичный для выбранного типа сервиса, и передает его на VPS по SSH.
4.  **Настройка защиты (Basic Auth)**: Если выбрано, создается `.htpasswd` файл на VPS для ограничения доступа к сервису по паролю.
5.  **Настройка SSL (для доменов)**: rProxy запрашивает сертификат Let's Encrypt через Certbot на вашем VPS.
6.  **Поднятие туннеля**: На роутере запускается процесс `autossh`, который пробрасывает порт с VPS на локальный сервис.

## 🛠 Варианты сервисов и их особенности

### 1. Веб-сервис (HTTP)
Предназначен для сайтов, панелей управления NAS, принтеров и т.д.
*   **Особенность**: rProxy автоматически транслирует заголовки `Host`, `X-Forwarded-For` и обрабатывает WebSockets.
*   **Путь**: `Браузер` → `VPS (Nginx)` → `SSH-Туннель` → `Роутер` → `Локальный сервис`.

### 2. Порт (TCP / Stream)
Используется для сервисов, не использующих HTTP (SSH, базы данных, игровые серверы) или для "сырого" проброса.
*   **Особенность**: Настраивается через блок `stream` в Nginx. Если используется домен, SSL терминируется на Nginx, и трафик идет защищенным до VPS.
*   **SSL для TCP**: rProxy поддерживает уникальную фичу — SSL поверх TCP через домен. Это позволяет, например, подключаться к SSH через защищенный TLS-порт.

### 3. Веб-терминал (ttyd)
Встроенная возможность rProxy для доступа к консоли роутера прямо из браузера.
*   **Особенность**: rProxy сам запускает процесс `ttyd` на роутере на свободном локальном порту (начиная с 7681) и пробрасывает его как HTTP-сервис.
*   **Безопасность**: Настоятельно рекомендуется использовать Basic Auth (шаг 7 мастера) для защиты терминала.

## 🔐 Проброс веб-интерфейса роутера (Keenetic)

Это самый важный и технически сложный момент. Веб-интерфейс Keenetic имеет встроенную защиту, которая может блокировать вход (ошибка пароля или бесконечная загрузка), если обращаться к нему через прокси.

### Как rProxy решает эту проблему:

При создании HTTP-сервиса для роутера (обычно на `192.168.1.1:80` или `127.0.0.1:80`), rProxy применяет специальные настройки в конфигурации Nginx на VPS:

1.  **Очистка заголовка Origin**: 
    ```nginx
    proxy_set_header Origin "";
    ```
    Это отключает проверку CSRF на стороне роутера. Без этого роутер может "сбрасывать" авторизацию, считая запрос подозрительным.
2.  **Подмена заголовка Host (Stealth Mode)**:
    Если доступ осуществляется по порту VPS (без домена), rProxy принудительно подставляет в заголовок `Host` локальный адрес роутера:
    ```nginx
    proxy_set_header Host "192.168.1.1";
    ```
    Роутер "видит", что запрос пришел как будто из локальной сети, и успешно принимает пароль.
3.  **Трансляция Cookies**:
    ```nginx
    proxy_cookie_domain "192.168.1.1" "$host";
    ```
    Это гарантирует, что сессионные куки роутера будут корректно приниматься вашим браузером.

> [!TIP]
> **Рекомендация**: При пробросе админки роутера всегда включайте **Basic Auth** в мастере добавления rProxy. Это создаст второй "кольцо" защиты перед тем, как злоумышленник вообще увидит стандартную страницу логина роутера.

## 🌐 Варианты доступа через VPS

### Вариант А: Домен + SSL (Рекомендуется)
Вы указываете домен (например, `router.myname.ru`).
*   **Плюсы**: Красивый адрес, безопасность (HTTPS), работает через стандартные порты (80/443).
*   **Требование**: Домен должен быть делегирован (A-запись) на IP вашего VPS.

### Вариант Б: Кастомный внешний порт
Вы указываете только порт (например, 26001).
*   **Плюсы**: Не нужен домен. Можно быстро "выкинуть" сервис наружу.
*   **Минусы**: Нет SSL (трафик идет по HTTP), адрес выглядит как `1.2.3.4:26001`.

## 🏥 Что делать, если пароль не принимается?

Если вы настроили проброс веб-интерфейса, но логин не проходит:

1.  **Проверьте адрес цели**: Убедитесь, что в rProxy указан именно тот IP и порт, на котором реально слушает веб-морда роутера (по умолчанию `192.168.1.1:80`).
2.  **Используйте домен**: Вход через домен с SSL работает стабильнее, так как современные браузеры неохотно работают с сессиями (Cookies) через незащищенный HTTP на нестандартных портах.
3.  **Очистите Кэш**: Иногда старые Cookies от других сессий KeenDNS или локальных входов могут конфликтовать. Попробуйте войти через "Инкогнито".
4.  **Проверьте Nginx**: Если вы редактировали конфиг вручную, убедитесь, что строки `proxy_set_header Origin "";` присутствуют в файле `/etc/nginx/sites-enabled/rproxy_ВАШ_СЕРВИС.conf` на вашем VPS.

## 🏗 Технические подробности и структура файлов

Для тех, кто хочет понимать, как всё устроено внутри:

### Структура файла сервиса (`/opt/etc/rproxy/services/NAME.conf`)
Каждый сервис описывается простым Shell-файлом с переменными. Эти файлы используются скриптом `rproxy` при запуске туннелей и в веб-интерфейсе.

### Параметры SSH-туннеля
rProxy использует `autossh` со следующими критическими параметрами:
*   `AUTOSSH_GATETIME=0`: Позволяет autossh не завершаться, если туннель не поднялся мгновенно.
*   `-M [PORT]`: Порт для мониторинга состояния туннеля (выбирается в районе 20000+).
*   `-R 0.0.0.0:[TUNNEL_PORT]:[TARGET_HOST]:[TARGET_PORT]`: Проброс порта с VPS на роутер. 

### Конфигурация Nginx на VPS
rProxy автоматически деплоит конфиги. Для веб-интерфейса роутера ключевым является сброс заголовка `Origin`, что позволяет проходить авторизацию без ошибок CSRF.

## 🐳 Команды управления (CLI)

Помимо интерактивного меню, вы можете управлять сервисами напрямую через консоль роутера:

*   `rproxy start [name]` — запустить указанный сервис (или все, если имя не указано).
*   `rproxy stop [name]` — остановить сервис.
*   `rproxy restart [name]` — перезапустить.
*   `rproxy status` — посмотреть текущий статус всех туннелей.
